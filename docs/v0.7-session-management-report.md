## 三、Session Architecture

### 数据库 Schema

新增一张表 `session_metadata`(通过 `CREATE TABLE IF NOT EXISTS` 自动迁移,新旧 DB 都兼容):

```sql
CREATE TABLE IF NOT EXISTS session_metadata (
  session_id   TEXT PRIMARY KEY,
  display_name TEXT,
  note         TEXT,
  tags         TEXT,    -- JSON array of strings
  pinned       INTEGER NOT NULL DEFAULT 0,
  created_at   TEXT NOT NULL,
  updated_at   TEXT NOT NULL
);
```

**关键设计点**:
- `session_id` 是 `sessions.id` 的外键等价关系(无显式 FK 约束,因为 `sessions` 是 collectors 拥有)
- `tags` 用 JSON 文本存储(去规范化,无 join 表)。在读路径上做 `parseTagsJson()` 兜底,损坏的 JSON → `[]`,不崩
- `pinned` 用 INTEGER 0/1(SQLite 无 boolean),前端展示时做 `!== 0`
- `upsert` 用 `INSERT ... ON CONFLICT DO UPDATE` 合并 patch,只更新传入的字段(保留未传入的字段)
- `created_at` 在首次写入时固化,`updated_at` 每次写入都更新

### 数据流

```
[Claude/Codex/Grok JSONL]                    [User in UI]
        │                                          │
        ▼                                          ▼
[Collectors]                               [PATCH /api/sessions-v2/:id/metadata]
        │                                          │
        ▼                                          ▼
   sessions 表  ←──── read-only ──────    session_metadata 表
        │                                          │
        └──────────────┬───────────────────────────┘
                       ▼
            GET /api/sessions-v2[?...]
            (LEFT JOIN + COUNT/SUM 子查询)
                       │
                       ▼
                Frontend Sessions / SessionDetail
```

sessions 表**永不修改**,所有用户定制走 session_metadata —— 这是 v0.7 的核心契约。

## 四、API 变化

新增 4 个路由,**全部 GET/PATCH,无副作用写入 sessions 表**:

| Method | Path | 用途 |
|--------|------|------|
| GET | `/api/sessions-v2` | 列表 + 搜索 + 过滤 |
| GET | `/api/sessions-v2/:id` | 单 session 完整 detail(基础 + metadata + git + usage + events + duration) |
| PATCH | `/api/sessions-v2/:id/metadata` | 改 displayName / note / tags / pinned |
| GET | `/api/sessions-v2/:id/resume` | 生成 resume 命令(纯字符串,**不执行**) |

### 列表查询参数(可组合)

| Param | 含义 | 示例 |
|-------|------|------|
| `agent` | 按 agent_id 过滤 | `agent=claude-code` |
| `project` | 按项目路径过滤 | `project=/p/agentos` |
| `search` | 在 `display_name \|\| title \|\| project` 中 LIKE | `search=agentos` |
| `status` | running/completed/failed/unknown | `status=completed` |
| `pinned` | `true`/`false` | `pinned=true` |
| `limit` | 1-1000,默认 500 | `limit=200` |

排序:**pinned 优先,然后 start_time DESC**。

### Resume 命令生成策略

`buildResumeCommand(agentType, externalId)` 纯函数,不执行任何 shell 命令:

| Agent | 命令模板 |
|-------|----------|
| `claude-code` | `claude --resume <id>` |
| `codex` | `codex resume <id>` |
| `grok` | `grok --resume <id>` |
| `gemini` / `hermes` / `custom` | `# Resume not yet supported for <agent>` |

- 用 `externalId`(不是 `agentId:externalId` 复合 id),命令可移植,不依赖 AgentOS
- 特殊字符(id 含 `'`)走单引号转义;常规 UUID 不加引号

### 兼容性

- 旧的 `/api/sessions` 和 `/api/sessions/:id` 路由**保留不变**,前端 v0.1-v0.6 的页面继续工作
- v2 的 detail 返回了 usage + events + git + metadata,一次 round-trip 拿全

五段式布局:

| 段 | 内容 |
|----|------|
| Header | 标题 + Agent + Status badge + 项目路径 + external id |
| 4 个统计卡 | Tokens / Cost / Events / Duration(自动格式化为 ms/s/m/h/d) |

## 十、当前限制

1. **search 不搜 tags**:设计上有意为之(tags 用 chip 过滤),但 UX 上用户可能期待。先不加,看反馈
2. **Resume 命令不支持 gemini/hermes/custom**:返回 placeholder,等这些 agent 暴露 CLI 再补
3. **tags 上限 32** + 自动去重 + 自动 trim:PATCH 路由的服务端硬限制,无法绕过
4. **没有批量操作**:不能一次改多个 session 的 metadata,UI 也只能逐个打开
5. **metadata 编辑后无 undo**:只显示 "Saved",没有 "Reset" 按钮回滚到上一次保存
6. **Pinned 没有数量上限**:用户可以 pin 任意多个 session,实际效果是"全部排前面",可能挤掉真正想找的
7. **没有 session 历史**:改名/改 tag 后只保留当前值,audit log 缺失
8. **搜索是 LIKE 匹配**:不支持 fuzzy / 正则 / 多关键词
9. **Resume 命令只在 macOS/Linux 测试**:Windows shell quote 行为不一样,需在真机验证
10. **SessionDetail 不分页**:usage/events 大数据量时一次返回全部(目前 limit 没限制)

## 附录:与 CC Switch 的差异

CC Switch 是 Claude Code 专用桌面 app 的 session 管理 UI,AgentOS v0.7 是 web 仪表盘。借鉴点:
- **搜索 UX**:list + search + filter 的范式,AgentOS 完全沿用
- **Rename / Tags / Pin**:三类核心用户定制,AgentOS 走 session_metadata 表实现
- **Resume 命令**:AgentOS 只生成不执行,确保安全

差异点(AgentOS 的额外能力):
- **跨 agent**:CC Switch 只管 Claude,AgentOS 同时管 Claude / Codex / Grok / Gemini / Hermes / custom
- **Git 集成**:每个 session 自带 commits 视图,CC Switch 没有
- **成本 / tokens**:每个 session 有完整 usage / cost 视图,CC Switch 只管元数据
- **Live + Timeline 双向跳转**:从 Live → Session 只需 2 次点击
